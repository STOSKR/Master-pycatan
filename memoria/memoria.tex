\section{Objetivo y estado actual}
El objetivo del trabajo es desarrollar un agente para Catan y evaluarlo de forma experimental, cubriendo:
\begin{itemize}
\item Parte 1: agente heuristico y benchmarks frente a random y agentes estandar.
\item Parte 2: integracion de LLM en dos decisiones del agente y analisis comparativo de prompts/modelos.
\end{itemize}

En esta entrega parcial se ha completado de forma solida la Parte 1 y se ha dejado implementada la infraestructura de la Parte 2 (pipeline y agente LLM), aunque faltan experimentos finales con UPV y Bedrock para cerrar esa parte con datos definitivos.

\section{Cambios implementados}
\subsection{Refactor de benchmarks}
Se unificaron los benchmarks en una infraestructura comun:
\begin{itemize}
\item \texttt{PyCatan/benchmark\_common.py}
\item wrappers: \texttt{benchmark\_vs\_random.py} y \texttt{benchmark\_vs\_agentes\_estandar.py}
\end{itemize}

Mejoras introducidas:
\begin{itemize}
\item ejecucion reproducible por semillas multiples;
\item misma estructura de metricas y CSV en ambos benchmarks;
\item configuracion por variables de entorno para agentes, seeds, partidas y workers;
\item soporte robusto en este entorno (fallback a \texttt{ThreadPoolExecutor} cuando \texttt{ProcessPoolExecutor} falla por permisos).
\end{itemize}

Campos homogeneos de salida: \texttt{wins}, \texttt{winrate}, \texttt{avg\_points}, \texttt{avg\_rank}, \texttt{fallback\_rate}, \texttt{provider}, \texttt{model}, \texttt{seed}.

\subsection{Agente base incremental}
Se implemento un nuevo tronco:
\begin{itemize}
\item \texttt{ShiyiBaseAgent}, que hereda de \texttt{RandomAgent}.
\end{itemize}

Este agente usa \textit{feature flags} por decision:
\begin{itemize}
\item \texttt{on\_game\_start}
\item \texttt{on\_build\_phase}
\item \texttt{on\_commerce\_phase}
\item \texttt{on\_moving\_thief}
\end{itemize}

Si una mejora esta desactivada o falla en runtime, el comportamiento vuelve a random (fallback seguro).

\subsection{Agente heuristico y agente LLM}
\begin{itemize}
\item \texttt{ShiyiHeuristicAgent} ahora extiende el tronco incremental.
\item \texttt{ShiyiLLMAgent} mantiene la logica LLM y ahora acepta configuracion por diccionario (facilita barridos de prompts/modelos desde benchmark).
\end{itemize}

\section{Diseno de heuristicas (Parte 1)}
El agente heuristico usa funciones de valor en:
\begin{itemize}
\item produccion esperada por nodo (ponderada por probabilidad de dado);
\item diversidad de recursos;
\item valor de expansion de carreteras;
\item objetivo dinamico de construccion (ciudad, pueblo, carretera, carta);
\item comercio con banca para cubrir escasez relevante.
\end{itemize}

La prioridad de construccion aplicada es:
\[
\text{ciudad} > \text{pueblo} > \text{carretera} > \text{carta}
\]

\section{Protocolo experimental}
\subsection{Pipeline iterativo}
Se implemento y ejecuto el pipeline:
\begin{itemize}
\item \texttt{iter0\_random\_baseline}
\item \texttt{iter1\_game\_start}
\item \texttt{iter2\_build\_phase}
\item \texttt{iter3\_commerce\_trade}
\item \texttt{iter4\_thief\_devcards}
\end{itemize}

Configuracion usada para el gate:
\begin{itemize}
\item seeds: \texttt{42, 31415, 2026}
\item 25 partidas por posicion (100 por seed)
\item criterio: media de \texttt{winrate} \(\geq 0.70\) vs random
\end{itemize}

Resultado consolidado: \texttt{PyCatan/benchmark\_iterative\_summary.csv}

\section{Analisis: por que aparece winrate=1.0 en iteracion 4}
No es un error de formato del CSV: en los experimentos mostrados se observan \texttt{wins=matches} y \texttt{avg\_rank=1.0000}.

Interpretacion mas plausible con los datos actuales:
\begin{itemize}
\item el agente random del simulador es muy debil frente a una politica heuristica de construccion/comercio bien orientada;
\item el salto principal viene de activar construccion y comercio (iter2 e iter3), no del modulo thief/development cards;
\item el resultado se mantiene en seeds nuevas, por lo que no parece sobreajuste a las 3 semillas iniciales.
\end{itemize}

Importante: este 100\% se da \textbf{contra random}. Contra estandar, en el estado actual, el rendimiento ya no es perfecto pero sigue siendo alto (alrededor de 90\% en este entorno y esta configuracion).

\section{Parte 2 (LLM): estado actual}
Se ha implementado la infraestructura necesaria:
\begin{itemize}

\item script de evaluacion ligera: \texttt{PyCatan/evaluate\_llm\_phase2.py};
\item soporte de proveedores \texttt{upv} y \texttt{bedrock} en el motor.
\end{itemize}

Pendiente para cerrar la rubrica de Parte 2:
\begin{itemize}
\item ejecutar y documentar al menos 3 modelos distintos con datos finales;
\item incluir comparativa de latencia/tokens/calidad/fallback entre prompts y proveedores.
\end{itemize}

\section{Uso de herramientas de IA}
Durante el desarrollo se usaron asistentes de IA para:
\begin{itemize}
\item refactorizar benchmarks y estandarizar metricas;
\item implementar el tronco incremental con fallback seguro;
\item preparar experimentacion iterativa y scripts de evaluacion LLM;
\item redactar y revisar analisis de resultados.
\end{itemize}

Las decisiones de diseno y la validacion se apoyaron en ejecuciones reales y revision manual de resultados CSV.

\section{Conclusiones}
\begin{enumerate}
\item La Parte 1 esta conseguida con evidencia experimental fuerte: el objetivo de \( \geq 70\% \) vs random se supera ampliamente.
\item El pipeline iterativo permite justificar que mejoras aportan mas rendimiento (especialmente build y commerce).
\item El resultado de 100\% vs random en iteraciones avanzadas es consistente en varias seeds, incluidas seeds fuera del conjunto inicial.
\item Contra agentes estandar, el rendimiento en el estado actual es alto, lo que confirma que las mejoras no solo explotan debilidades triviales del random.
\item La Parte 2 esta implementada a nivel tecnico, pero requiere cerrar la experimentacion con modelos UPV/Bedrock para completar la practica al 100\%.
\end{enumerate}
